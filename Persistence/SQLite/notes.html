Persistence Notes

What are transactions?

Transactions are a set of operations that are executed together as a single unit. They ensure that either all the operations are completed successfully or none of them take effect. This prevents data inconsistency, especially in systems handling sensitive information, such as banking applications.

What are ACID Properties?

ACID properties define the key characteristics of a reliable database transaction:

  • Atomicity: Ensures that a transaction is all-or-nothing. If any part of the transaction fails, the entire operation is rolled back.
  • Consistency: Guarantees that a transaction brings the database from one valid state to another, maintaining integrity constraints.
  • Isolation: Ensures that concurrent transactions do not interfere with each other, preventing issues like dirty reads and lost updates.
  • Durability: Ensures that once a transaction is committed, it remains in the database even if a system failure occurs.

Suppose you do not have transactions. Is that system useful? Why?

Without transactions, the system may still function, but it would be highly unreliable. If a failure occurs mid-operation, partial data may be saved, leading to corruption and inconsistencies. This is especially problematic in applications requiring high data integrity, such as financial systems.

What properties does your file system have?

File systems provide a way to store, organize, and manage data efficiently. Most modern file systems, like NTFS or ext4, support features like journaling (which helps recover data after crashes), permissions (to control access), and caching (to optimize performance). However, file systems do not inherently offer ACID compliance like databases.

Suppose you do not have "A" in ACID? What happens? When is it ok?

Without Atomicity, partial transactions can be committed, leading to data corruption. Imagine transferring money between two bank accounts—if the debit operation succeeds but the credit operation fails, money could disappear. However, in scenarios like logging or telemetry data collection, atomicity may not be critical, as partial entries still provide useful insights.

Suppose you do not have "C" in ACID? What happens? When is it ok?

Lack of Consistency means the database might enter an invalid state. For example, if an e-commerce website allows an item’s stock to go negative due to inconsistent updates, it could cause operational problems. However, in distributed systems using eventual consistency (like NoSQL databases), temporary inconsistencies are acceptable as long as the system stabilizes over time.

Suppose you do not have "I" in ACID? What happens? When is it ok?

Without Isolation, concurrent transactions might interfere with each other, leading to issues like dirty reads, where one transaction sees uncommitted changes from another. This could be problematic in online booking systems, where multiple users might double-book the same seat. However, in high-speed analytics where performance matters more than strict correctness, reducing isolation levels can improve efficiency.

Suppose you do not have "D" in ACID? What happens? When is it ok?

Without Durability, data might be lost if the system crashes after a transaction is committed. This is unacceptable for applications like financial transactions or medical records. However, for temporary caches or in-memory databases where persistence isn’t critical, durability may not be a strict requirement.
